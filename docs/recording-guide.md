# 会话录制功能使用指南

## 概述

会话录制功能提供两种录制模式：**后端录制**和**前端录制**。两种模式各有适用场景，选择哪种取决于你的需求。

---

## 两种模式对比

| | 后端录制 | 前端录制 |
|---|---|---|
| **录制原理** | 服务端通过 Puppeteer/Playwright 渲染页面并录屏 | 浏览器端使用 rrweb 库捕获 DOM 事件流 |
| **录制内容** | 视频文件（MP4） | DOM 事件流 → 后端合成为视频 |
| **对用户影响** | 无感知，无性能影响 | 录制期间消耗客户端 CPU/内存 |
| **适用场景** | 定时巡检、无人值守的页面监控 | 用户操作回放、Bug 复现、用户行为分析 |
| **交互录制** | 不支持（只能录制页面渲染结果） | 支持（鼠标移动、点击、输入等） |
| **视频质量** | 高（服务端渲染，帧率可控） | 取决于事件密度和后端合成 |
| **网络依赖** | 仅需一个 URL | 需要持续将事件发送至后端 |

---

## 一、后端录制

### 使用方式

1. 进入 **录制管理** 页面（侧边栏 → 会话录制 → 录制管理），切换到 **后端录制** 标签页。
2. 填写录制参数：
   - **目标 URL**（必填）：要录制的网页地址，例如 `https://example.com`
   - **录制时长**（5-300 秒）：录制持续的时间
   - **画质**：高/中/低，画质越高视频文件越大
   - **帧率**（1-30 FPS）：帧率越高视频越流畅，但生成时间更长
3. 点击 **开始录制**，任务提交至后端。
4. 页面会显示任务状态卡片，包含流水线进度：
   - 录制中 → 排队中 → 处理中 → 完成
5. 任务完成后，前往 **视频回放** 页面查看生成的视频。

### 工作流程

```
用户填写参数 → POST /recording/start → 后端创建任务
                                         ↓
                               Puppeteer 打开目标 URL
                                         ↓
                               录制指定时长后停止
                                         ↓
                               FFmpeg 合成视频文件
                                         ↓
前端轮询 GET /recording/status/:id ← 状态变更
                                         ↓
                         视频生成完毕 → 视频回放页面播放
```

### 注意事项

- 目标 URL 必须可从服务端访问（外网或内网）
- 如果页面需要登录，后端录制无法携带登录态（需在后端配置 Cookie 注入）
- 录制时长越长，视频生成等待时间越长

## 二、前端录制

### 使用方式

1. 进入 **录制管理** 页面，切换到 **前端录制** 标签页。
2. 点击 **开始录制** 按钮。
3. 在当前页面进行你需要的操作（点击、输入、滚动等）。
4. 录制过程中可以看到实时统计：
   - **已录制**：录制持续的秒数
   - **事件数**：已捕获的 DOM 事件总数
   - **已发送批次**：已发送到后端的事件批次
5. 点击 **停止录制** 结束录制。
6. 后端将事件流合成为视频，完成后可点击 **查看视频** 链接或在视频回放页面查看。

### 工作流程

```
用户点击"开始录制" → POST /recording/start（创建任务）
                         ↓
              rrweb 开始捕获 DOM 事件（mousemove、click、input、scroll 等）
                         ↓
              每 5 秒批量发送事件 → POST /recording/events
                         ↓
用户点击"停止录制" → 发送剩余事件 → POST /recording/stop
                         ↓
              后端将事件流重放并合成视频（FFmpeg）
                         ↓
              前端轮询状态 → 视频生成完毕
```

### 性能优化措施

前端录制已内置以下优化，避免浏览器卡顿：

| 优化项 | 说明 |
|---|---|
| **鼠标采样** | mousemove 事件每 500ms 最多采集一次，避免高频事件洪水 |
| **滚动采样** | scroll 事件每 150ms 最多采集一次 |
| **事件计数节流** | 每秒更新一次 UI 计数器，而非每次事件触发时都重新渲染 |
| **分批次发送** | 单次最多发送 500 个事件，大数组拆分为多个请求，避免 JSON 序列化阻塞 UI 线程 |
| **停止反馈** | 点击停止后显示加载状态，避免用户误以为页面卡死 |

### 注意事项

- **录制期间不要刷新页面**，否则录制中断且已采集的事件无法恢复
- 录制时间建议控制在 **5 分钟以内**，长时间录制会积累大量事件数据
- 如果页面 DOM 结构极其复杂，初始事件体积可能较大（rrweb 会记录完整 DOM 快照）
- 前端录制依赖 `rrweb` 库，首次使用时会动态加载，加载期间不阻塞页面

## 三、视频回放

1. 进入 **视频回放** 页面（侧边栏 → 会话录制 → 视频回放）。
2. 页面展示最近 50 条录制任务，包含 Task ID、目标 URL、状态、事件数、创建时间。
3. 状态为 **已完成** 的任务，点击 **播放** 按钮可在线观看视频。
4. 点击 **详情** 可查看任务的完整信息。
5. 存在未完成任务时，页面每 5 秒自动刷新列表。

## 四、常见问题

### Q: 后端录制报错 "无法访问目标 URL"

检查目标 URL 是否可从服务端网络访问。如果目标地址在内网，确保录制服务部署在同一网络环境。

### Q: 前端录制点击"停止录制"后页面卡顿

已通过事件采样和分批发送优化。如果仍然卡顿，建议缩短录制时长或降低页面复杂度。停止过程中会显示 "正在停止..." 状态，等待数据传输完成即可。

### Q: 前端录制的事件数一直为 0，但已录制时间在增长

可能是 `rrweb` 库加载失败。打开浏览器控制台查看是否有加载错误，检查网络是否能访问 npm CDN。

### Q: 视频生成完成后在视频回放页面无法播放

检查视频流地址是否可访问。视频流通过 `GET /recording/video/:taskId` 提供，确认后端服务正在运行且端口可访问。

### Q: 前端录制和后端录制可以同时使用吗？

可以。两者是独立的录制任务，互不影响。

## 五、API 接口速查

| 方法 | 端点 | 说明 |
|---|---|---|
| POST | `/recording/start` | 创建录制任务 |
| POST | `/recording/events` | 上传录制事件批次（前端录制专用） |
| POST | `/recording/stop` | 停止录制任务 |
| GET | `/recording/status/:taskId` | 查询任务状态与详情 |
| GET | `/recording/list` | 获取最近 50 条任务列表 |
| GET | `/recording/video/:taskId` | 获取视频流（用于 `<video>` 播放） |
